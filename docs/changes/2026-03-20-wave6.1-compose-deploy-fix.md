# Wave 6.1 Compose Deploy Fix

Date: 2026-03-20
Scope: production deploy script selection of Docker Compose implementation
Change type: deploy-path hardening

## What Changed

- changed `scripts/deploy_prod.sh` to resolve the Docker Compose command on the target host before the deploy begins
- made `docker compose` the preferred command when the Compose V2 plugin is installed
- allowed `docker-compose` only when it is actually backed by Compose V2
- blocked legacy `docker-compose` v1 with a hard error and explicit operator guidance
- added deploy-context logging for host, runner context, selected compose command, compose version, and docker version
- documented the verified server-side execution path and fix in `docs/state/Welle_6.1_compose_deploy_fix_summary.md`

## Why It Changed

The production workflow runs on a self-hosted GitHub Actions runner installed on the target server itself, and `scripts/deploy_prod.sh` was still invoking `docker-compose` directly.

Live verification showed that the target server currently has:

- no `docker compose` plugin command
- `docker-compose 1.29.2`
- Docker `28.2.2`

That makes the deploy path depend on Compose v1 on the server, which is the most likely source of the observed `KeyError: 'ContainerConfig'` failure.

## Affected Files

- scripts/deploy_prod.sh
- docs/state/Welle_6.1_compose_deploy_fix_summary.md
- docs/state/instance-structure-unification-plan.md

## Operational Impact

- production deploy now refuses to proceed when only Compose v1 is available on the target server
- once the server has the Compose V2 plugin, the deploy script will prefer `docker compose`
- deploy logs now make the actual server-side compose path explicit before any stack action is attempted

## Compatibility Notes

- no container topology was changed
- no compose file content was changed
- no database, analytics, or BlackLab behavior was changed
- no destructive cleanup behavior was added

## Follow-Up

- install the Docker Compose V2 plugin on the production server so `docker compose` is available
- rerun the deploy after the server-side compose tooling is upgraded
