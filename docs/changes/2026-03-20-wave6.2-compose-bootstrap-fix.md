# Wave 6.2 Compose Bootstrap Fix

Date: 2026-03-20
Scope: production deploy workflow bootstrap before invoking `scripts/deploy_prod.sh`
Change type: deploy-path hardening

## What Changed

- changed `.github/workflows/deploy.yml` so the self-hosted production workflow updates `/srv/webapps/corapan/app` to `GITHUB_SHA` before invoking `scripts/deploy_prod.sh`
- documented the forensic findings and bootstrap root cause in `docs/state/Welle_6.2_compose_guard_forensics_summary.md`

## Why It Changed

Wave 6.2 forensics showed that the workflow already ran:

- `cd /srv/webapps/corapan/app`
- `bash scripts/deploy_prod.sh`

But the app checkout was not refreshed before that script was launched. That means deploy-script changes are not self-applying on the first run: an older on-disk `scripts/deploy_prod.sh` can still start first, then enter the old `docker-compose` v1 path, and only afterwards fetch newer code.

Bootstrapping the target checkout to `GITHUB_SHA` before the script starts removes that gap.

## Affected Files

- .github/workflows/deploy.yml
- docs/state/Welle_6.2_compose_guard_forensics_summary.md
- docs/state/instance-structure-unification-plan.md

## Operational Impact

- the production deploy workflow now executes the deploy script version from the triggering commit, not whichever copy happened to already be present on disk
- future fixes to `scripts/deploy_prod.sh` take effect on the first relevant deploy run
- if the target host still only has Compose v1, the deploy will now fail early with the Wave 6.1 guard instead of falling through to a later compose-v1 traceback

## Compatibility Notes

- no service topology changed
- no runtime paths changed
- no data, database, or BlackLab behavior changed
- no destructive cleanup behavior was added
